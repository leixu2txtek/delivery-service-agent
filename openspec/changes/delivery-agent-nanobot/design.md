## Context

现场实施人员遇到问题时，当前依赖微信群和人工反馈，存在以下痛点：
- 问题响应慢，依赖技术专家在线状态
- 知识分散在飞书文档中，检索困难
- 新人缺乏经验，遇到问题无从下手

本系统基于 nanobot（LLM 驱动的 Web 自动化 Agent 框架）构建交付智能体，集成飞书 MCP 实现文档检索，通过 H5 页面提供移动端友好交互。

**技术约束：**
- 采用 OpenAI 协议兼容的大模型（支持 Agent 能力）
- 飞书知识库实时检索，不使用本地向量库
- 部署在云服务，支持移动端访问
- 7 天会话自动清理，控制存储成本

## Goals / Non-Goals

**Goals:**
- 实施人员可通过 H5 页面自然语言提问，快速获得 FAQ 解答
- 系统能识别需要引导诊断的问题，分步收集信息并给出建议
- 无法解决的问题自动转人工，发送群消息并创建待办任务
- 支持历史会话查看，7 天后自动清理
- 手机号+密码登录，保障访问安全

**Non-Goals:**
- 不执行实际的配置变更（只提供指导）
- 不接外部业务系统（如客户配置系统、日志系统）
- 不支持语音输入/输出
- 不实现复杂的权限管理（只区分登录/未登录）

## Decisions

### 1. 飞书文档检索采用 MCP 而非本地向量库
**决策：** 使用 `lark-openapi-mcp` 直接调用飞书 API 检索文档

**理由：**
- 保证知识库实时性，与飞书文档保持一致
- 减少向量库维护成本
- 飞书搜索 API 对中文支持较好

**替代方案：** 定期同步飞书文档到本地向量库（Milvus/Qdrant）
- 放弃原因：同步延迟、需要额外维护向量数据库

### 2. 前端采用 H5 而非小程序/APP
**决策：** 开发响应式 H5 页面，部署到 CDN

**理由：**
- 跨平台，无需适配多个小程序平台
- 部署简单，更新即时生效
- 实施人员手机浏览器即可访问

**替代方案：** 微信小程序
- 放弃原因：需要企业资质、审核流程、无法快速迭代

### 3. 实时通信采用 WebSocket + HTTP 混合
**决策：** WebSocket 用于对话消息，HTTP 用于其他 API

**理由：**
- 支持流式输出（打字机效果）
- 保持连接状态，减少重复认证
- 心跳机制检测在线状态

**替代方案：** 纯 HTTP SSE
- 放弃原因：SSE 在移动端弱网环境下重连复杂

### 4. 会话状态存储在 Redis
**决策：** 使用 Redis 存储对话上下文和会话状态

**理由：**
- 支持 TTL 自动过期，方便实现 7 天清理
- 性能高，适合高频读写
- 可横向扩展

**替代方案：** 纯数据库存储
- 放弃原因：需要手动清理过期数据，性能较差

### 5. 大模型选择支持 OpenAI 协议的模型
**决策：** 接口层兼容 OpenAI API，支持 GPT-4/Claude/国产模型

**理由：**
- 灵活性高，可根据成本/效果切换模型
- 支持 Function Calling 实现 Agent 能力
- 国内部署可选择文心一言/通义千问等

**替代方案：** 本地部署开源模型
- 放弃原因：需要 GPU 资源、维护成本高

### 6. 转人工触发策略
**决策：** 三条件触发：用户主动输入、诊断流程结束未解决、AI 置信度低

**理由：**
- 平衡用户体验和人工成本
- 避免过度依赖人工
- 关键信息已收集，减少重复沟通

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|-----|------|---------|
| 飞书 API 调用延迟高 | 用户体验差 | 增加 loading 状态、异步检索、设置超时回退 |
| 大模型 API 不稳定 | 服务不可用 | 多模型兜底、本地缓存常见问题回答 |
| 飞书机器人 OAuth 过期 | 转人工功能失效 | Token 自动刷新、过期告警、手动重新授权流程 |
| 历史会话数据增长 | 存储成本上升 | 7 天 TTL 自动清理、只保留关键字段 |
| 并发量突增 | 服务崩溃 | 限流保护、队列机制、水平扩展 |

## Migration Plan

**部署步骤：**
1. 准备云服务器、域名、HTTPS 证书
2. 部署 Redis、MySQL
3. 部署飞书 MCP 服务，完成 OAuth 认证
4. 部署 nanobot 智能体服务
5. 部署 H5 前端到 CDN
6. 创建飞书机器人，配置群通知
7. 初始化管理员账号
8. 小规模试点（5-10 人），收集反馈
9. 逐步推广到全部实施人员

**回滚策略：**
- 保留微信群作为 fallback
- 服务异常时显示"系统维护中，请使用微信群"
- 数据库备份每日执行
